# 产品设计哲学

## 通过标准化减少选择困难

**核心问题：**

> "选择困难 <-- 设计标准化套餐
>
> 游客没有时间、精力和耐性去筛选和定制和服套餐。
>
> 商家没有时间、精力和方法来管理种类繁多的库存和预约。"

## 当前租赁套餐模型

平台采用**预打包套餐**模式，而非自定义组装模式：

**RentalPlan 数据结构：**

```typescript
model RentalPlan {
  // 核心标识
  name         String    // 清晰的套餐名称（例如："经典女士套餐"）
  category     Category  // LADIES | MENS | COUPLE | FAMILY | GROUP | SPECIAL

  // 固定价格
  price        Int       // 单一价格点（以分为单位）
  originalPrice Int?     // 可选，用于显示折扣

  // 固定时长
  duration     Int       // 包含的小时数（例如：4、6、8 小时）

  // 预定义包含项
  includes     String[]  // 包含内容（例如：["和服租赁", "专业着装", "发型设计"]）

  // 精选展示
  imageUrl     String?   // 专业照片
  description  String    // 清晰说明
  tags         Tag[]     // 可筛选属性

  // 运营相关
  isActive     Boolean   // 上架/下架开关
  isCampaign   Boolean   // 活动价格
  isFeatured   Boolean   // 首页推荐
}
```

## 如何减少决策负担

### 对用户（游客/旅行者）：

#### 1. 快速对比
所有同类别套餐显示相同的信息结构：
- 价格、时长和包含项一目了然
- 没有隐藏费用或意外附加费
- 可按分类（女士/男士/情侣）和地区筛选

#### 2. 透明的预期
`includes[]` 数组清楚说明包含什么：
- 无需了解和服类型或配饰知识
- 避免从 50+ 个选项中选择的决策疲劳
- "和服租赁 + 专业着装 + 发型设计"一目了然

#### 3. 通过标准建立信任
套餐由商家精选，而非用户自行组装：
- 商家预先选择协调的和服和配饰
- 默认包含专业造型建议
- 降低"选错了"的焦虑

#### 4. 快速预订流程
单页结账，最少步骤：
- 加入购物车 → 选择日期/时间 → 确认
- 没有 10 个决策点的多步向导
- 支持游客结账（无需强制注册）

### 对商家：

#### 1. 简化库存管理
- 创建 5-10 个标准套餐，而非管理 100+ 个单品
- 每个套餐代表他们可以可靠准备的"包"
- `includes[]` 作为运营清单

#### 2. 标准化定价
- 固定价格点，消除议价
- `originalPrice` 支持可控折扣
- `isCampaign` 标记用于促销期

#### 3. 可预测的运营
- `duration` 字段设定明确的时间预期
- `category` 帮助资源分配（女士 vs 男士更衣室）
- `isActive` 允许即时上架/下架而无需删除

#### 4. 市场就绪
- 平台可通过审批强制执行质量标准
- 佣金计算简单直接（`price` 的百分比）
- 所有商家呈现一致

## 设计权衡与讨论点

### ✅ 当前优势：

#### 1. 简单胜于灵活
- 用户做 2-3 个决策（分类、日期、店铺）vs 20+ 项选择
- 80% 的用户想要的是"给我一个好的和服体验"
- 借鉴成功模式：Airbnb "即时预订"、Apple 产品分级

#### 2. 运营可扩展性
- 商家可以用标准化套餐扩展到每天 100+ 订单
- 员工培训更简单（"准备套餐 A" vs "检查 15 个定制选项"）
- 减少沟通错误

#### 3. 价格透明
- 单一价格点减少结账放弃率
- 没有附加费用的"诱饵和转换"
- 竞争比较简单明了

### ⚠️ 当前限制与开放性问题：

#### 1. 有限的个性化

- **问题**：我们是否应该允许"套餐定制"（例如：加 ¥500 添加额外配饰）？
- **权衡**：灵活性 vs 保持简单
- **当前方法**：使用 `tags` 作为个性化信号（例如："适合拍照"、"传统风格"）

#### 2. 和服选择控制

- **问题**：用户是否想在到达前预选特定的和服设计？
- **权衡**：选择控制 vs 物流复杂性
- **当前方法**：`imageUrl` 显示代表性和服，实际选择在店内进行
- **未来考虑**：在预订时添加 `BookingKimono` 选择（架构中已存在）

#### 3. 附加项 vs 全包

- **问题**：高利润项目（专业摄影、接发）应该是单独附加项吗？
- **权衡**：收入优化 vs 决策疲劳
- **当前方法**：核心项目在 `includes[]` 中，高级服务可以是可选的
- **架构支持**：购物车系统中存在 `addOns` 字段但未充分利用

#### 4. 多套餐预订

- **问题**：如果情侣一起预订，他们应该选择 2 个单独套餐还是 1 个"情侣套餐"？
- **权衡**：灵活性（不同偏好）vs 简单性（一次结账）
- **当前方法**：购物车系统都支持，但鼓励情侣套餐

#### 5. 商家套餐泛滥

- **问题**：一个商家应该被允许创建多少套餐？
- **风险**：太多套餐重新制造选择困难
- **当前方法**：无强制限制，但 UI 以紧凑网格显示套餐
- **考虑**：建议每个分类 3-5 个套餐，使用 `isFeatured` 突出畅销品

#### 6. 基于时间的定价

- **问题**：高峰时段（周末、假期）应该有动态定价吗？
- **权衡**：收入优化 vs 价格透明
- **当前方法**：`isCampaign` 用于促销期，但没有高峰定价
- **架构限制**：单一 `price` 字段，没有基于日期的定价规则

## 推荐最佳实践

### 平台运营：

- 鼓励商家创建 3-8 个总套餐（而非 30+）
- 每个套餐应该有明显差异（价格/时长/包含项）
- 使用 `tags` 进行软个性化（搜索/筛选），而非套餐繁殖

### 商家入驻：

- 提供套餐模板："基础套餐"、"豪华套餐"、"全日体验"
- 指导："首次租赁和服的人想要什么？"
- 不鼓励：单个和服选择、单点定价

### 用户体验：

- 首页展示 8-12 个精选套餐（横向滚动）
- 分类页面以网格显示 12-20 个套餐
- 筛选是辅助性的，非必需的（地区、价格范围、标签）
- 套餐详情页强调 `includes[]`，用视觉对勾标记

## 未来考虑

### 1. 引导推荐

AI 聊天机器人根据以下条件推荐套餐：
- 场合（拍照、节日、休闲）
- 团体组成（单人、情侣、家庭）
- 预算和时间限制

### 2. 智能捆绑

自动建议互补套餐：
- "您的朋友也选择了套餐，作为情侣可节省 ¥500"
- 购物车级别优化 vs 单个套餐选择

### 3. 套餐模板

管理员精选的商家"入门包"：
- 也减少商家的决策疲劳
- 确保平台质量一致性

### 4. 性能指标

跟踪哪些套餐转化最好：
- `currentBookings` 字段已存在
- 使用数据建议商家简化表现不佳的套餐
