  四、"积木模型"设计理念

  三层可升级组件：

  | 层级  | 名称              | 内容           | 升级路径         |

  | 主菜  | The Main Kimono | 基础/蕾丝/访问着/振袖 | 基础→蕾丝→访问着→振袖 |

  五、首页改造方向

  从"套餐列表"到"场景漏斗"

  1. 首页：只展示6个主题卡片，不展示具体套餐

  Airbnb 的设计策略：分层展示 + 横向滚动

  Airbnb 并没有选择"只显示分类"或"只显示具体服务"，而是采用了混合策略：

  1. 顶部：搜索框 + 快速分类标签（可横向滚动）
  2. 内容区：多个横向滚动的卡片"走马灯"，每个走马灯对应一个主题
  3. 卡片：直接展示具体服务，但按主题分组

  ---
  方案对比分析

  | 维度      | 只显示场景卡片     | 直接显示套餐      | Airbnb混合模式      |
  |---------|-------------|-------------|-----------------|
  | 首屏信息密度  | 低（6张卡片）     | 高（12-20个套餐） | 中高（分组+横滚）       |
  | 决策路径    | 2步（选主题→选套餐） | 1步（直接选套餐）   | 1.5步（可直接选，也可筛选） |
  | 用户教育    | 强（主题页讲故事）   | 弱（用户自己判断）   | 中（分组标题引导）       |
  | 选择困难症   | ✅ 缓解        | ❌ 加重        | ✅ 缓解            |
  | 高意向用户体验 | ❌ 多一步点击     | ✅ 直达        | ✅ 直达            |
  | SEO/分享  | ❌ 具体套餐埋太深   | ✅ 套餐直接可见    | ✅ 套餐直接可见        |

  | 纯场景卡片    | 品牌建设期、需要强用户教育  |

  Legacy 套餐处理策略

  问题：76 个现有套餐中有 20+ 个有订单关联

  | 处理方式                 | 优点   | 缺点                       |
  |----------------------|------|--------------------------|
  | 硬删除                  | 干净   | 破坏 BookingItem 外键，历史数据丢失 |
  | 软删除 (isActive=false) | 保留历史 | 数据库膨胀，slug 冲突风险          |
  | 归档表                  | 完全隔离 | 需要新建表，迁移复杂               |
  | 打标签 + 隐藏             | 灵活   | 需要前端配合过滤                 |

  推荐：渐进式迁移 + 软删除

  第一阶段：并行运行
  ├─ 创建 Theme 模型
  ├─ 创建 21 个新套餐（按 spreadsheet）
  ├─ 新套餐关联 Theme
  ├─ 旧套餐保持 themeId = null
  └─ 首页优先展示有 themeId 的套餐

  第二阶段：切换
  ├─ 将旧套餐 isActive = false
  ├─ 但保留 BookingItem 关联
  └─ 历史订单页面仍可查看

  第三阶段：清理（可选）
  ├─ 1-2 个月后
  ├─ 确认无问题
  └─ 可选择归档或保留
